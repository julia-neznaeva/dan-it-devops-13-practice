A) KERNEL
----------
----------

Ядро (kernel) є центральним компонентом архітектури операційної системи Linux. Саме kernel працює в kernel space, безпосередньо взаємодіє з апаратним забезпеченням і контролює всі критичні ресурси системи. Будь-яка робота з процесором, оперативною пам’яттю, дисками, мережею або пристроями введення-виведення відбувається виключно через ядро. Усі прикладні програми та системні сервіси можуть звертатися до апаратури лише опосередковано — через системні виклики, які обробляє kernel.

АРХІТЕКТУРА ЯДРА LINUX

Linux використовує монолітну архітектуру ядра з підтримкою динамічних модулів. Усі ключові підсистеми — планувальник процесів, менеджер пам’яті, файлові системи, мережевий стек та драйвери пристроїв — працюють у kernel space. Драйвери реалізовані у вигляді модулів ядра (.ko), які можуть динамічно завантажуватися та вивантажуватися без перезавантаження системи.

Такий підхід забезпечує високу продуктивність і мінімальні накладні витрати, оскільки основна логіка виконується безпосередньо в ядрі, без додаткових рівнів абстракції. Водночас модульність дозволяє зберігати гнучкість і розширюваність системи.

МОДУЛЬНІСТЬ І ДРАЙВЕРИ

Драйвери в Linux є частиною ядра, але можуть керуватися динамічно. За допомогою інструментів modprobe та rmmod можливе завантаження й вивантаження драйверів під час роботи системи. Це дозволяє адаптувати систему до нового апаратного забезпечення без зупинки або перезавантаження.

Модульна архітектура ядра сприяє широкій підтримці апаратури та швидкому впровадженню нових драйверів. Саме ця властивість зробила Linux придатним для використання в найрізноманітніших середовищах — від серверів і хмарних платформ до вбудованих систем.

Ліцензія та модель розвитку

Linux kernel поширюється під ліцензією GPLv2. Весь код ядра є відкритим, а будь-які модифікації зобов’язані публікуватися. Така модель розвитку забезпечує прозорість, колективну перевірку коду та швидке виявлення помилок.

Відкрита ліцензія сприяла формуванню великої спільноти розробників і зробила Linux kernel основою для значної частини сучасної інфраструктури — серверних систем, хмарних платформ, embedded-рішень та мобільних операційних систем.

ГНУЧКІСТЬ І КАСТОМІЗАЦІЯ

Linux kernel можна збирати й конфігурувати під конкретні потреби. Можливо вимикати непотрібні підсистеми, оптимізувати ядро під конкретне апаратне забезпечення або сценарій використання. Один і той самий kernel може працювати як у мінімальному середовищі з обмеженими ресурсами, так і в масштабних обчислювальних кластерах.

Така гнучкість робить Linux kernel універсальним компонентом, який не прив’язаний до одного типу пристроїв або задач.

ІЗОЛЯЦІЯ ТА КЕРУВАННЯ РЕСУРСАМИ

Linux kernel містить вбудовані механізми ізоляції та контролю ресурсів. Namespaces дозволяють створювати ізольовані простори для процесів, файлових систем, мережі, ідентифікаторів користувачів та міжпроцесної взаємодії. Control groups (cgroups) забезпечують обмеження використання процесорного часу, оперативної пам’яті, дискового вводу-виводу та кількості процесів.

Ці механізми працюють на рівні ядра і застосовуються до звичайних процесів, не потребуючи спеціальної віртуалізації.

ЯК DOCKER ВИКОРИСТОВУЄ LINUX KERNEL

Docker не емулює апаратне забезпечення і не має власного ядра. Він працює у user space і напряму використовує можливості Linux kernel. Під час виконання команди docker run Docker Engine через системні виклики просить kernel створити ізольований процес.

Ядро створює окремі namespaces, у межах яких процес має власний простір процесів, файлову систему, мережевий інтерфейс, hostname та простір користувачів. Паралельно kernel застосовує cgroups для обмеження використання ресурсів. Файлова система контейнера формується за допомогою overlayfs, яка об’єднує read-only шари образу з writable-шаром.

Безпека забезпечується на рівні ядра через capabilities, seccomp та, за наявності, механізми мандатного контролю доступу. Після цього kernel виконує системні виклики clone() та execve() і запускає процес. Контейнер у Linux — це звичайний процес, але ізольований, обмежений і захищений.

KERNEL ЯК ФУНДАМЕНТ DOCKER І KUBERNETES

Контейнерні технології повністю залежать від можливостей Linux kernel. Namespaces, cgroups, overlayfs і seccomp є частиною ядра і не можуть бути реалізовані поза ним. Саме тому Docker на не-Linux платформах працює через запуск Linux kernel у віртуальному середовищі.

Kubernetes не взаємодіє з kernel напряму. Він керує контейнерами на рівні кластера, передаючи завдання container runtime, який уже звертається до Linux kernel. У цій моделі kernel створює середовище виконання, Docker або інший runtime запускає процеси, а Kubernetes керує їх масштабом і життєвим циклом.

ПІДСУМОК РОЗДІЛУ

Linux kernel є фундаментом усієї операційної системи. Він створює процеси, ізолює їх, обмежує ресурси та забезпечує безпеку. Саме завдяки можливостям ядра Linux став основою для сучасних серверних систем, контейнерних платформ і хмарної інфраструктури.


b) Libraries
-------------
-------------

Бібліотеки (libraries) є важливим компонентом архітектури операційної системи Linux і виконують роль проміжного шару між прикладними програмами та ядром. На відміну від kernel, бібліотеки працюють у user space і не мають прямого доступу до апаратного забезпечення. Їх основне призначення — надати програмам зручний і стандартизований програмний інтерфейс (API) для використання можливостей операційної системи.

У типовій архітектурі Linux шлях виконання операцій виглядає так: прикладна програма викликає функцію бібліотеки, бібліотека за потреби викликає системний виклик, kernel виконує реальну роботу з ресурсами, після чого результат повертається назад у програму. Таким чином, бібліотеки є необхідною ланкою між кодом програми та ядром.

РОЛЬ БІБЛІОТЕК В АРХІТЕКТУРІ ОС

Без бібліотек кожна програма була б змушена самостійно реалізовувати роботу з файлами, мережею, пам’яттю та потоками виконання, використовуючи низькорівневі системні виклики. Це призвело б до дублювання коду, підвищеної складності та зростання кількості помилок.

Бібліотеки вирішують цю проблему, виконуючи кілька ключових функцій:

стандартизують доступ до можливостей операційної системи;

приховують складність взаємодії з kernel;

зменшують обсяг коду прикладних програм;

підвищують переносимість програм між різними Linux-системами;

сприяють безпеці, обмежуючи прямий доступ програм до ядра.

СИСТЕМНІ БІБЛІОТЕКИ

Основу бібліотечного шару Linux складають системні бібліотеки. Найважливішою з них є стандартна бібліотека C — libc. Саме через libc більшість програм отримують доступ до файлової системи, пам’яті, процесів і мережі.

Функції на кшталт fopen(), malloc(), printf() або pthread_create() є частиною API бібліотек. Усередині вони можуть виконувати додаткову логіку, а за потреби викликати відповідні системні виклики ядра, наприклад open(), read(), write() або clone().

Таким чином, бібліотека не виконує реальну роботу з апаратурою, а лише формує коректний запит до kernel.

ТИПИ БІБЛІОТЕК У LINUX

У Linux існує кілька типів бібліотек, які відрізняються способом підключення та використання.

Динамічні бібліотеки підвантажуються під час запуску програми та можуть спільно використовуватися багатьма процесами. У Linux вони мають розширення .so. Такий підхід зменшує споживання пам’яті та дозволяє оновлювати бібліотеки без перекомпіляції програм.

Статичні бібліотеки вбудовуються безпосередньо в програму під час компіляції. Вони мають розширення .a. У цьому випадку виконуваний файл стає більшим, але не залежить від наявності бібліотек у системі.

Окрему категорію складають бібліотеки мов програмування вищого рівня, таких як Python, Java або JavaScript. Хоча вони знаходяться на більш абстрактному рівні, в кінцевому підсумку вони також використовують системні бібліотеки і через них звертаються до ядра.

МЕЖА МІЖ LIBRARY І KERNEL

Принциповою є межа між бібліотеками та ядром. Kernel працює в kernel space, має повний доступ до апаратного забезпечення і відповідає за безпеку та стабільність системи. Бібліотеки ж працюють у user space, не мають прямого доступу до ресурсів і не можуть керувати процесором, пам’яттю чи пристроями напряму.

Бібліотека не здатна виконати операцію з апаратурою без участі ядра. Єдиним шляхом у kernel є системний виклик, і бібліотеки виступають зручним механізмом формування таких викликів.

ПРИКЛАД: ВІДКРИТТЯ ФАЙЛУ

Типовий приклад взаємодії бібліотеки з ядром — відкриття файлу. Коли програма викликає fopen("file.txt", "r"), вона звертається до функції з libc. Усередині бібліотека аналізує аргументи, готує необхідні структури даних і викликає системний виклик open. Kernel перевіряє права доступу, працює з файловою системою і повертає результат назад через бібліотеку у програму.

Цей приклад демонструє, що бібліотеки не замінюють kernel, а лише спрощують доступ до його можливостей.

БІБЛІОТЕКИ В КОНТЕЙНЕРАХ

У контейнерних середовищах бібліотеки є частиною образу контейнера. Різні контейнери можуть містити різні версії libc або інших бібліотек, але всі вони використовують один і той самий Linux kernel хост-системи.

Це означає, що бібліотеки формують користувацьке середовище контейнера, тоді як kernel залишається спільним і виконує всі системні операції. Такий підхід є однією з ключових відмінностей контейнерів від віртуальних машин.

Взаємодія людини з бібліотеками

Людина не може використовувати бібліотеку напряму так само, як системну утиліту. Бібліотека не має власного CLI і не запускається як окремий процес. Її API складається з функцій, які можуть бути викликані лише з програмного коду.

Водночас користувач опосередковано постійно використовує бібліотеки — через системні утиліти, прикладні програми або інтерпретатори мов програмування. Наприклад, команда ls використовує libc, а виклики бібліотек у Python здійснюються через інтерпретатор, який є окремою програмою.

ПІДСУМОК РОЗДІЛУ

Бібліотеки є ключовим елементом архітектури Linux, що з’єднує прикладні програми з ядром. Вони не керують ресурсами і не взаємодіють з апаратурою напряму, але роблять використання можливостей операційної системи зручним, стандартизованим і безпечним. Саме завдяки бібліотекам розробка програм у Linux є масштабованою та переносимою.


C) SYSTEM UTILITIES
---------------------
---------------------

System utilities (системні утиліти) є користувацькими програмами, призначеними для керування операційною системою Linux та взаємодії з її внутрішніми механізмами. Вони працюють у user space і не є частиною ядра, проте саме через них людина отримує практичний доступ до можливостей операційної системи.

На відміну від бібліотек, які призначені для використання з програмного коду, system utilities є готовими виконуваними програмами, що запускаються як окремі процеси і приймають команди у вигляді текстового вводу.

МІСЦЕ SYSTEM UTILITIES В АРХІТЕКТУРІ ОС

В архітектурі Linux system utilities розташовуються між користувачем і нижчими програмними рівнями. Типовий шлях виконання виглядає так: користувач вводить команду в терміналі, shell обробляє введений текст і запускає відповідну утиліту, утиліта викликає функції бібліотек, бібліотеки виконують системні виклики, а kernel здійснює реальну роботу з ресурсами системи.

Таким чином, system utilities не взаємодіють з апаратним забезпеченням напряму. Вони лише формують запити до ядра у зручній для людини формі.

ПРИЗНАЧЕННЯ ТА РОЛЬ СИСТЕМНИХ УТИЛІТ

Без system utilities kernel був би практично недоступним для користувача. Будь-яка адміністративна дія вимагала б написання власних програм, що напряму викликають бібліотеки або системні виклики.

System utilities виконують роль інтерфейсу між людиною та операційною системою. Вони дозволяють:

керувати файлами і каталогами;

переглядати та контролювати процеси;

налаштовувати права доступу;

працювати з мережею;

керувати системними сервісами;

отримувати інформацію про стан ядра та системи загалом.

Завдяки системним утилітам Linux стає придатним для інтерактивної роботи, адміністрування та автоматизації.

ОСНОВНІ КАТЕГОРІЇ SYSTEM UTILITIES

Системні утиліти охоплюють практично всі аспекти керування системою. До утиліт для роботи з файловою системою належать команди ls, cp, mv, rm, chmod, chown, df і du. Вони дозволяють переглядати вміст каталогів, керувати файлами та контролювати використання дискового простору.

Для керування процесами використовуються утиліти ps, top, htop, kill, nice і renice. Вони не керують процесами самостійно, а отримують інформацію від kernel і надсилають йому сигнали.

Окрему групу складають утиліти для роботи з користувачами та правами доступу, зокрема who, whoami, id, su і sudo. Мережеві утиліти (ip, ss, ping, traceroute, curl, wget) забезпечують доступ до мережевого стеку через стандартні інтерфейси ядра.

До системних інструментів також належать утиліти для роботи з ядром і системою загалом, такі як uname, lsmod, modprobe, dmesg, free та uptime. Управління сервісами здійснюється через systemctl і journalctl, які є інтерфейсом до systemd, а не до kernel напряму.

SYSTEM UTILITIES І KERNEL: ПРИНЦИПОВА МЕЖА

Попри те, що system utilities використовуються для керування системою, вони не мають привілейованого доступу до апаратного забезпечення. Усі утиліти працюють у user space і підпорядковуються тим самим правилам безпеки, що й звичайні програми.

Kernel залишається єдиним компонентом, який має прямий доступ до ресурсів. System utilities лише запитують інформацію або ініціюють дії, а kernel вирішує, чи дозволити ці операції.

ПРИКЛАД: ВИКОНАННЯ КОМАНДИ PS

Показовим прикладом роботи системних утиліт є команда ps. Вона не зберігає власний список процесів і не “знає” про них самостійно. Під час виконання ps читає дані з файлової системи /proc, яка є інтерфейсом, що надається ядром. Kernel формує інформацію про процеси, а утиліта ps лише інтерпретує і форматуює ці дані для відображення користувачу.

Цей приклад демонструє, що system utilities не дублюють логіку ядра, а лише представляють її результати у зручному вигляді.

SYSTEM UTILITIES І КОНТЕЙНЕРИ

У контейнерних середовищах системні утиліти є частиною образу контейнера. Вони працюють з тим самим Linux kernel, що й хост-система, але завдяки namespaces бачать інший, ізольований простір. Саме тому команда ps усередині контейнера показує лише процеси цього контейнера, а файлові утиліти працюють з іншою ієрархією каталогів.

Це підкреслює, що поведінка system utilities визначається не самими утилітами, а середовищем, яке створює kernel.

SYSTEM UTILITIES, CLI І БІБЛІОТЕКИ

System utilities тісно пов’язані з командним інтерфейсом (CLI). Користувач вводить команди в shell, shell запускає відповідні утиліти, а ті у свою чергу використовують бібліотеки для виклику системних функцій.

Таким чином, system utilities відрізняються від бібліотек тим, що є самостійними програмами для користувача, тоді як бібліотеки призначені для використання з програмного коду. Обидва ці компоненти працюють у user space, але виконують різні ролі в архітектурі системи.

ПІДСУМОК РОЗДІЛУ

System utilities є основним засобом взаємодії людини з операційною системою Linux. Вони перетворюють можливості kernel у доступний командний інтерфейс, не виконуючи низькорівневу роботу самостійно. У поєднанні з бібліотеками та ядром системні утиліти формують завершену і практичну архітектуру ОС Linux, придатну для адміністрування, автоматизації та повсякденної роботи.
